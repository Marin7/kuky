# Implementation Plan: Multi-Language Support for Static UI Text

**Branch**: `008-language-support` | **Date**: 2026-06-12 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/008-language-support/spec.md`

## Summary

Add Spanish / Romanian / English language switching to all static UI text across the Kuky front-end. Language is stored in `localStorage` (client-side only); no URL routing changes. The server always renders in Spanish; the client switches on hydration via `react-i18next`. A `LanguageSwitcher` dropdown (flag + code) is added to the existing `SiteHeader`. Dynamic/teacher-entered content is explicitly excluded.

## Technical Context

**Language/Version**: TypeScript 5.8 (strict), React 19

**Primary Dependencies**: `react-i18next` + `i18next` (to be installed); `@radix-ui/react-dropdown-menu` (already in project via Shadcn `DropdownMenu`)

**Storage**: `localStorage` key `kuky-lang` (`'es' | 'ro' | 'en'`)

**Testing**: Manual validation via `quickstart.md`; TypeScript strict mode catches key typos at build time

**Target Platform**: Browser (client-side only for language switching); SSR renders Spanish default

**Project Type**: Web application — front-end only change

**Performance Goals**: Language switch completes in < 2 seconds (from SC-001); no full page reload

**Constraints**: No URL changes; no back-end changes; only static UI text in scope

**Scale/Scope**: ~8 routes + ~30 components; ~300–500 translation keys estimated across 3 languages

### Architecture Flow

```
App boot (client)
  └── __root.tsx useEffect
        ├── read localStorage.getItem('kuky-lang')
        ├── if valid: i18next.changeLanguage(stored)
        └── set <html lang={lang}>

User clicks LanguageSwitcher
  └── useLanguage.setLanguage(lang)
        ├── i18next.changeLanguage(lang)
        ├── localStorage.setItem('kuky-lang', lang)
        └── (React re-renders all t() calls automatically)
```

### Key Interfaces

**`useLanguage` hook**:
```typescript
// Shape only
interface UseLanguage {
  language: 'es' | 'ro' | 'en'
  setLanguage: (lang: 'es' | 'ro' | 'en') => void
}
```

**`LANGUAGES` constant** (defined in `i18n/index.ts`):
```typescript
// Shape only
interface Language {
  code: 'es' | 'ro' | 'en'
  flag: string    // e.g. '🇪🇸'
  label: string   // e.g. 'ES'
}
```

**Translation file shape** (see [data-model.md](data-model.md) for full key tree):
```typescript
// es.ts exports `const esDict = { nav: { home: 'Inicio', ... }, ... }`
// ro.ts and en.ts follow the same shape (partial allowed — missing keys fall back to es)
```

### Component Integration Pattern

Every component that currently renders hardcoded Spanish text:
1. Imports `useTranslation` from `react-i18next`
2. Calls `const { t } = useTranslation()`
3. Replaces each literal string with `t('namespace.key')`

Dynamic data (homework titles, exercise text, resource names) remains as-is — passed as props, rendered directly.

### SSR Consideration

`i18next` is initialised synchronously before the React tree renders. On the server, it always starts with `lng: 'es'`. On the client, the hydration `useEffect` in `__root.tsx` reads `localStorage` and calls `changeLanguage` — this may cause a brief render with Spanish text before switching. This is acceptable per the confirmed architecture (Option B in clarification Q1).

To suppress the SSR/client mismatch warning (if it appears), `i18next` can be initialised with `initImmediate: false`.
