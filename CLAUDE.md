# Kuky — Español con Paula

Personal website for Paula, a Spanish teacher targeting Romanian students.
A full-stack application with a React SSR frontend and a Java Spring Boot REST API.

## Tech stack

### Front-end

| Layer | Technology |
|-------|-----------|
| Framework | React 19 + TanStack Start 1.x (SSR) + TanStack Router (file-based) |
| Styling | TailwindCSS 4.x + Shadcn UI (Radix UI primitives) |
| Build | Vite 7 via `@lovable.dev/vite-tanstack-config` |
| Package manager | npm (bun preferred — install from https://bun.sh if available) |
| Language | TypeScript 5.x strict |
| Runtime | **Node.js 22+ or 24+** required (Vite 7 minimum; Node 20.17 is too old) |

### Back-end

| Layer | Technology |
|-------|-----------|
| Language | Java 26 |
| Framework | Spring Boot 4.0.x + Spring Security + plain JDBC (NamedParameterJdbcTemplate) |
| Database | PostgreSQL 18 (local: pgAdmin; production: managed PostgreSQL) |
| Migrations | Flyway 11.x — **no auto-configuration in Spring Boot 4**; manually wired via `config/FlywayConfig.java` |
| Auth | JWT in HTTP-only SameSite=Lax cookies, 7-day rolling session |
| Password hashing | BCrypt cost factor 12 |
| Rate limiting | Custom sliding-window counter (10 attempts / 15 min per IP) |
| Email | JavaMailSender + Mailpit (local SMTP capture) |
| Build | Gradle |

```bash
./gradlew test                                              # run tests only
```

## Local dev setup

1. **PostgreSQL 18** — running via pgAdmin. Create the database and user once:
   ```sql
   CREATE DATABASE kuky_dev;
   CREATE USER kuky WITH PASSWORD 'kuky';
   GRANT ALL PRIVILEGES ON DATABASE kuky_dev TO kuky;
   GRANT CREATE ON SCHEMA public TO kuky;
   ALTER USER kuky WITH SUPERUSER;   -- simplest for local dev
   ```
2. **Mailpit** — SMTP capture for email testing. Run `mailpit` (port 1025 SMTP, http://localhost:8025 web UI).
3. **Backend** — `./gradlew bootRun --args='--spring.profiles.active=local'` — Flyway runs migrations on first start via `FlywayConfig` (Spring Boot 4 has no Flyway auto-configuration).
4. **Frontend** — `npm run dev` from `front-end/`.

## Key implementation notes

### Front-end
- **Image imports**: Use standard Vite imports (`import url from "@/assets/foo.jpg"`),
  NOT Lovable-specific `.asset.json` pattern — that only works inside Lovable's cloud.
- **Routing**: File-based via TanStack Router. `routeTree.gen.ts` is auto-generated;
  never edit it manually. Add a new page by creating `src/routes/<name>.tsx`.
- **Active nav links**: Use `activeProps` + `activeOptions={{ exact: true }}` on `<Link>`
  for the home route (`/`); other routes don't need `exact`.
- **Mobile nav**: `SiteHeader.tsx` shows a hamburger button on `md:hidden` that opens a
  Shadcn `Sheet` with the same five nav links.
- **SSR**: TanStack Start renders all pages server-side. Client-only code must live in
  event handlers or `useEffect`.
- **Fonts**: Fraunces (display/headings via `font-display` class) and Inter (body) loaded
  from Google Fonts in `__root.tsx`.
- **Auth (client-side)**: `cuenta.tsx` manages auth state via `useEffect` + `getMe()`.
  All API calls go through `src/lib/auth.ts` with `credentials: 'include'` for cookie handling.
- **Scheduling UI**: Calendly-style two-panel layout — `CalendarPicker` (2-week grid, available days highlighted) on the left, `TimeSlotList` (open slots for the selected day) on the right. Clicking a slot opens `BookingDialog`. All date/time labels use `"es"` locale (Spanish names for days and months).
- **Cross-component refresh**: `reservas.tsx` wires two refs — `scheduleRefreshRef` (passed to `ScheduleView`) and `myBookingsRefreshRef` (passed to `MyBookings`). After a booking succeeds, `ScheduleView` calls both so the calendar slot flips to BOOKED and "Mis clases" appears immediately without a page reload.

### Back-end
- **Auth cookie**: Named `auth-token`, `HttpOnly`, `SameSite=Lax`, `maxAge=604800` (7 days).
  Rolling refresh: every authenticated request resets the MaxAge (handled in `JwtCookieAuthenticationFilter`).
- **Email normalisation**: Stored and compared in lowercase (FR-018).
- **Password reset tokens**: UUID v4, 1-hour expiry, single-use. Previous unused tokens
  invalidated on new request and on consumption.
- **Rate limiting**: `LoginRateLimiter` — sliding window, 10 attempts per 15 min per IP,
  in-memory `ConcurrentHashMap<String, Deque<Long>>`. No external dependency.
- **Error codes**: All API errors return `{"error":"ERROR_CODE","message":"..."}`.
  Codes defined in `contracts/api.md` and mapped in `GlobalExceptionHandler`.
- **Local mail**: When Mailpit is not running, `EmailService` logs the reset URL to the
  console (`WARN EmailService — password reset URL for ...`).
- **Flyway**: Spring Boot 4.0 removed Flyway auto-configuration entirely. Migrations are triggered manually by `FlywayConfig.java` which creates a `Flyway` bean, calls `migrate()`, and relies on Spring's dependency ordering to ensure it runs before any repository queries.
- **CORS**: `CorsConfig` exposes a `CorsConfigurationSource` bean (not `WebMvcConfigurer`). `SecurityConfig` calls `.cors(Customizer.withDefaults())` so Spring Security processes CORS before auth checks — required for authenticated endpoints (e.g. `POST /api/v1/bookings`) to correctly handle OPTIONS preflights.
- **MeetingProvider selection**: `MeetingProviderConfig` is a `@Configuration` factory that checks `app.zoom.client-id` at runtime — blank → `StubMeetingProvider` (returns `zoom.invalid` URL, logs WARN); non-blank → `ZoomMeetingProvider`. Do **not** use `@ConditionalOnProperty` for this; Spring Boot 4 treats an empty-string property as "present", which caused both beans to register simultaneously.
- **Scheduling rules**: Virtual slots generated by `AvailabilityService` — no `availability_slots` table. Rules applied in order: weekends (Sat/Sun) → UNAVAILABLE; lunch break (12:00–14:00 teacher time, configurable) → UNAVAILABLE; within lead-time window or in the past → UNAVAILABLE; confirmed booking exists → BOOKED; otherwise → OPEN. All rules also enforced in `validateBookable` to block direct API abuse.

The back-end reads these at startup (set in your deployment environment):

| Variable | Description |
|----------|-------------|

| `DB_USERNAME` | Database username |
| `DB_PASSWORD` | Database password |
| `APP_JWT_SECRET` | HS256 signing key, at least 32 characters |
| `CORS_ALLOWED_ORIGIN` | Frontend origin, e.g. `https://kuky.es` |
| `FRONTEND_BASE_URL` | Used in password reset emails, e.g. `https://kuky.es` |
| `MAIL_HOST` | SMTP host (defaults to `localhost`) |
| `MAIL_PORT` | SMTP port (defaults to `1025`) |
| `MAIL_FROM` | From address (defaults to `noreply@kuky.es`) |
| `TEACHER_EMAIL` | Paula's email for booking/cancellation notifications (defaults to `noreply@kuky.es`) |
| `ZOOM_ACCOUNT_ID` | Zoom Server-to-Server OAuth account ID |
| `ZOOM_CLIENT_ID` | Zoom Server-to-Server OAuth client ID |
| `ZOOM_CLIENT_SECRET` | Zoom Server-to-Server OAuth client secret |
| `ZOOM_USER_ID` | Zoom user who hosts meetings (defaults to `me`) |

The following scheduling defaults can be overridden via `app.scheduling.*` in YAML (no env-var wiring needed unless you want runtime control):

| YAML key | Default | Description |
|----------|---------|-------------|
| `app.scheduling.teacher-timezone` | `Europe/Madrid` | Timezone for slot generation and display |
| `app.scheduling.day-start` | `09:00` | First slot start time (teacher local time) |
| `app.scheduling.day-end` | `18:00` | Last slot end boundary (teacher local time) |
| `app.scheduling.lunch-break-start` | `12:00` | Lunch break start — slots always UNAVAILABLE |
| `app.scheduling.lunch-break-end` | `14:00` | Lunch break end |
| `app.scheduling.class-duration-minutes` | `60` | Slot duration and step size |
| `app.scheduling.min-lead-hours` | `24` | Minimum hours ahead required to book |
| `app.scheduling.cancel-cutoff-hours` | `24` | Minimum hours ahead required to cancel |

### Teacher backoffice (`/panel`) — admin-only

- **Admin role**: `users.role` is `STUDENT` or `ADMIN` (migration `V6`). The role is carried as a JWT
  claim and enforced by `SecurityConfig` (`.requestMatchers("/api/v1/admin/**").hasRole("ADMIN")`);
  guests get 401, authenticated non-admins get 403 `ACCESS_DENIED` (JSON handlers in `SecurityConfig`).
  `/me` (and login/register/reset) return `role` so the frontend gates the `Panel` nav entry and the
  `/panel` route. **`AdminBootstrap`** (a `CommandLineRunner`) promotes the account whose email equals
  `app.scheduling.teacher-email` to `ADMIN` on startup — the role lands in the JWT on that user's **next login**.
- **Availability is now data-driven**: the public `/reservas` schedule is generated from teacher-managed
  `availability_rules` (weekly pattern) + `availability_exceptions` (date BLOCK/OPEN) — migration `V7`,
  seeded to reproduce the old Mon–Fri 09–18 schedule with the lunch gap. The hard-coded
  `day-start`/`day-end`/`lunch-break-*` config is gone; `AvailabilityService` still layers the existing
  min-lead-time, no-past, and confirmed-booking `BOOKED` safeguards on top. Saving availability never
  cancels bookings; the save response returns confirmed bookings now outside availability as a warning.
- **Homework is per-student**: a `homework_targets` join table (migration `V8`) replaces the old
  shared-to-all model — a student sees a homework item only if assigned to them. The `V5` seeded
  assignments have no targets, so they are now unshared drafts.
- **Presentations**: teacher-authored decks (`presentations` / `presentation_slides` / `presentation_shares`,
  migration `V9`) with uploaded images stored in an `images` BYTEA table and served via
  `GET /api/v1/images/{id}`. A deck is visible to a student only if shared (else 404). Image uploads are
  validated for type (jpeg/png/webp) and size (≤ 2 MB).
